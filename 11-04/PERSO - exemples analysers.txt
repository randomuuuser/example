# Contexte unifié pour tous les exemples
**Tâche** : classifier des commentaires de clients (positif/négatif/neutre)  
**Modèle** : BERT fine-tuné sur des avis produits  
**Données** : 10 000 commentaires étiquetés (60% positifs, 25% négatifs, 15% neutres)

---

## 1. ClassificationAnalyser

### Métrique 1 : Accuracy
**Explication intuitive** : L'accuracy est la proportion de prédictions correctes parmi toutes les prédictions. Elle répond à la question : "Dans combien de cas le modèle a-t-il vu juste, tous types de commentaires confondus ?" Une accuracy de 0.85 signifie que sur 100 commentaires, le modèle en classe correctement 85.

**Application concrète** : Sur 2000 commentaires de test, le modèle prédit correctement 1700 étiquettes.
- **Calcul** : 1700 / 2000 = **0.85** (ou 85%)
- **Interprétation** : Le modèle est correct dans 85% des cas. Mais attention, si 85% des commentaires sont positifs, un modèle qui prédit toujours "positif" aurait 85% d'accuracy sans rien apprendre → d'où l'importance du F1.

### Métrique 2 : F1 macro
**Explication intuitive** : Le F1 macro fait la moyenne des F1 de chaque classe, en donnant le même poids à la classe minoritaire qu'à la majoritaire. Il répond : "Comment le modèle performe-t-il en moyenne sur chaque type de commentaire, indépendamment de leur fréquence ?" Un F1 macro de 0.81 signifie que la performance moyenne par classe est de 81%.

**Application concrète** : Après prédiction sur 2000 commentaires de test :
- Positif : précision 0.92, rappel 0.89 → F1 = 2*(0.92*0.89)/(0.92+0.89) = 0.90
- Négatif : précision 0.85, rappel 0.80 → F1 = 0.82
- Neutre : précision 0.72, rappel 0.70 → F1 = 0.71

**Calcul** : (0.90 + 0.82 + 0.71) / 3 = **0.81**
- **Interprétation** : Le modèle performe très bien sur le positif (90%), bien sur le négatif (82%), mais nettement moins sur le neutre (71%). La moyenne de 81% masque cette disparité.

### Métrique 3 : F1 micro
**Explication intuitive** : Le F1 micro aggrège tous les vrais positifs, faux positifs et faux négatifs de toutes les classes avant de calculer le F1. Il répond : "Quelle est la performance globale du modèle en considérant chaque prédiction individuelle ?" Il est dominé par les classes majoritaires.

**Application concrète** :
- Total vrais positifs (somme sur toutes classes) = 1650
- Total faux positifs = 180
- Total faux négatifs = 170
- Précision micro = 1650/(1650+180) = 0.90
- Rappel micro = 1650/(1650+170) = 0.91
- **F1 micro** = 2*(0.90*0.91)/(0.90+0.91) = **0.905**
- **Interprétation** : Le F1 micro (0.905) est plus élevé que le F1 macro (0.81) car la classe majoritaire (positif, 60% des données) performe très bien et tire la moyenne vers le haut.

### Visualisation 1 : Matrice de confusion normalisée
**Explication intuitive** : Un tableau qui montre, pour chaque classe réelle (lignes), comment les prédictions se répartissent (colonnes). Les valeurs sont des proportions (0 à 1) qui s'additionnent à 1 par ligne. Elle répond : "Quand un commentaire est réellement de classe X, quelle est la probabilité qu'il soit classé comme Y ?"

**Application concrète** :
```
            Prédit
           Pos  Neut  Neg
Réel Pos  0.88 0.09 0.03
     Neut 0.15 0.70 0.15
     Neg  0.04 0.08 0.88
```
- **Interprétation ligne par ligne** :
  - Quand un commentaire est réellement Positif (1ère ligne) : 88% sont bien classés positifs, 9% sont classés neutres (erreur), 3% négatifs.
  - Quand un commentaire est réellement Neutre (2e ligne) : seulement 70% sont bien classés neutres ; 15% sont classés positifs, 15% négatifs → la classe neutre est très ambigüe.
  - Quand un commentaire est réellement Négatif (3e ligne) : 88% bien classés, 8% neutres, 4% positifs.

### Visualisation 2 : Per-label F1 bar chart
**Explication intuitive** : Un graphique à barres montrant le F1 de chaque classe individuellement. Il permet de voir d'un coup d'œil quelles classes posent problème.

**Application concrète** :
- Barre Positif : hauteur 0.90 (verte)
- Barre Négatif : hauteur 0.82 (orange)
- Barre Neutre : hauteur 0.71 (rouge)
- **Interprétation** : Visualisation immédiate que la classe "Neutre" est le maillon faible. On peut décider d'investiguer pourquoi (ambiguïté sémantique, déséquilibre, etc.)

### 1.1 SequenceClassificationAnalyser

#### Métrique 1 : ROC-AUC (One-vs-Rest)
**Explication intuitive** : L'AUC mesure la capacité du modèle à séparer une classe donnée de toutes les autres, quel que soit le seuil de décision choisi. Une AUC de 0.94 signifie que si on prend aléatoirement un commentaire positif et un commentaire non-positif, le modèle donnera un score plus élevé au positif dans 94% des cas.

**Application concrète** : Pour la classe "positif"
- On calcule pour chaque seuil de probabilité le Taux de Vrais Positifs vs. Taux de Faux Positifs.
- L'aire sous cette courbe = **0.94**
- **Interprétation** : Excellent pouvoir discriminant. Le modèle distingue très bien les commentaires positifs des autres.

#### Métrique 2 : Expected Calibration Error (ECE)
**Explication intuitive** : L'ECE mesure l'écart moyen entre la confiance annoncée par le modèle et sa précision réelle. Un ECE de 0.03 signifie qu'en moyenne, la confiance prédite diffère de la précision réelle de 3 points de pourcentage. Plus l'ECE est proche de 0, mieux le modèle est calibré.

**Application concrète** :
- Regroupement par déciles de confiance prédite (0-10%, 10-20%, ...)
- Décile 90-100% : confiance moyenne prédite = 0.94, précision réelle = 0.89
- Écart = 0.05 pour ce décile
- Moyenne des écarts sur tous les déciles = **0.03**
- **Interprétation** : Le modèle est bien calibré. Quand il dit "je suis sûr à 94%", il a raison dans 89% des cas → léger sur-confiance (3 points d'écart en moyenne).

#### Visualisation 1 : Normalised confusion matrix (déjà vu ci-dessus)

#### Visualisation 2 : Predicted probability distribution per class
**Explication intuitive** : Un histogramme qui montre, pour chaque classe, comment se distribuent les probabilités prédites par le modèle (de 0 à 1). Cela permet de voir si le modèle est sûr de lui ou hésitant.

**Application concrète** : Pour la classe "positif" :
- Histogramme bimodal : pic à 0.05-0.15 (commentaires clairement non-positifs) et pic à 0.85-0.98 (commentaires clairement positifs)
- Très peu de probabilités entre 0.3 et 0.7
- **Interprétation** : Le modèle est rarement indécis sur la classe positive. Il sépare bien le positif du non-positif.

#### Visualisation 3 : Per-label precision/recall/F1 bar chart
**Explication intuitive** : Un graphique à barres groupées montrant pour chaque classe la précision (des prédictions de cette classe, combien sont correctes), le rappel (des vrais éléments de cette classe, combien sont trouvés) et le F1 (moyenne harmonique des deux).

**Application concrète** :
```
Classe    Précision  Rappel    F1
Positif   0.92       0.89      0.90
Négatif   0.85       0.80      0.82
Neutre    0.72       0.70      0.71
```
- **Interprétation** : Pour le neutre, la précision (72%) signifie que quand le modèle prédit "neutre", il a raison dans 72% des cas. Le rappel (70%) signifie que sur tous les vrais neutres, le modèle en trouve 70%. Les deux sont faibles → le modèle sous-performe systématiquement sur cette classe.

#### Visualisation 4 : Reliability diagram (calibration curve)
**Explication intuitive** : Un graphique qui compare la confiance prédite (axe X) à la précision réelle (axe Y). La diagonale représente une calibration parfaite. Les écarts à la diagonale montrent où le modèle est sur-confiant (points sous la diagonale) ou sous-confiant (points au-dessus).

**Application concrète** :
- 10 bins de 0 à 1
- Bin [0.9-1.0] : confiance moyenne prédite = 0.94, précision réelle = 0.89 → point sous la diagonale
- Bin [0.5-0.6] : confiance moyenne prédite = 0.55, précision réelle = 0.58 → point au-dessus de la diagonale
- **Interprétation** : Le modèle est légèrement sur-confiant pour ses prédictions les plus sûres, et légèrement sous-confiant pour ses prédictions moyennement sûres.

### 1.1.1 NLIAnalyser

#### Métrique 1 : Per-relation F1
**Explication intuitive** : Pour chaque relation (entailment, contradiction, neutral), le F1 mesure comment le modèle reconnaît cette relation spécifique. Un F1 de 0.73 pour "neutral" signifie que la détection des paires sans relation implicative est médiocre.

**Application concrète** (inférence naturelle sur des paires commentaire-règle) :
- Entailment (le commentaire implique la règle) : F1 = 0.87
- Contradiction : F1 = 0.82
- Neutral : F1 = 0.73
- **Interprétation** : Le modèle excelle à détecter quand un commentaire implique une règle (87%) et quand il la contredit (82%), mais il a du mal avec les cas où il n'y a ni implication ni contradiction (73%).

#### Métrique 2 : Label frequency bias
**Explication intuitive** : Cette métrique compare la distribution des prédictions à la distribution réelle des classes. Un écart de +9 points pour "contradiction" signifie que le modèle prédit cette relation 9% plus souvent qu'elle n'apparaît réellement.

**Application concrète** :
- Distribution gold (réelle) : Entailment=45%, Contradiction=22%, Neutral=33%
- Distribution prédite : Entailment=40%, Contradiction=31%, Neutral=29%
- Biais = prédit - gold = [-5%, +9%, -4%]
- **Interprétation** : Le modèle a un fort biais vers la contradiction (9 points de trop) et un biais négatif vers le neutral (4 points de moins). Il sur-prédit les contradictions.

#### Visualisation 1 : Per-relation confusion matrix
**Explication intuitive** : Une matrice 3x3 qui montre, pour chaque relation réelle (lignes), comment les prédictions se répartissent (colonnes).

**Application concrète** :
```
           Prédit
           Ent  Cont  Neut
Réel Ent   0.85 0.08 0.07
     Cont  0.06 0.82 0.12
     Neut  0.10 0.17 0.73
```
- **Interprétation** : Les neutres réels sont confondus avec entailment (10%) et surtout contradiction (17%). Cela explique le biais : le modèle transforme les neutres en contradictions.

#### Visualisation 2 : Predicted vs. gold label distribution bar chart
**Explication intuitive** : Un graphique à barres côte à côte montrant la proportion de chaque classe dans les données réelles (gold) vs. dans les prédictions.

**Application concrète** :
- Deux barres par classe (gold en bleu, prédit en orange)
- Entailment : 45% gold vs. 40% prédit (orange plus basse)
- Contradiction : 22% gold vs. 31% prédit (orange plus haute)
- Neutral : 33% gold vs. 29% prédit (orange plus basse)
- **Interprétation** : Visualisation immédiate du biais : la barre orange de "contradiction" dépasse nettement la barre bleue.

### 1.2 TokenClassificationAnalyser

#### Métrique 1 : Token-level accuracy
**Explication intuitive** : Proportion de tokens (mots ou sous-mots) correctement étiquetés. Une accuracy de 0.91 signifie que sur 100 tokens, 91 reçoivent le bon tag.

**Application concrète** : Annotation des mentions de produits sur 5000 tokens.
- Tokens correctement taggés = 4550
- **Token-level accuracy** = 4550/5000 = **0.91**
- **Interprétation** : 9% des tokens sont mal taggés. Mais cette métrique masque les erreurs de tagging (confondre B-PRODUCT et I-PRODUCT).

#### Métrique 2 : Per-label F1
**Explication intuitive** : Pour chaque type de tag (B-PRODUCT, I-PRODUCT, O), le F1 mesure la performance du modèle sur ce tag spécifique. Un F1 de 0.92 pour B-PRODUCT signifie que le modèle identifie très bien les débuts d'entités.

**Application concrète** :
- B-PRODUCT (début d'entité produit) : précision 0.94, rappel 0.91 → F1 = 0.92
- I-PRODUCT (interne d'entité) : précision 0.89, rappel 0.87 → F1 = 0.88
- O (hors entité) : précision 0.96, rappel 0.97 → F1 = 0.965
- **Interprétation** : Le modèle performe très bien sur les tokens hors entité (96.5%), un peu moins sur les tokens internes (88%), ce qui est normal car les entités sont plus rares.

#### Visualisation 1 : Per-tag confusion matrix
**Explication intuitive** : Une matrice montrant pour chaque tag réel (lignes) la distribution des tags prédits (colonnes). Elle révèle quelles paires de tags sont souvent confondues.

**Application concrète** :
```
              B-PROD  I-PROD  O
B-PROD réel    0.91   0.05   0.04
I-PROD réel    0.07   0.88   0.05
O réel         0.01   0.01   0.98
```
- **Interprétation** : 5% des B-PROD sont prédits comme I-PROD (erreur de tagging) et 7% des I-PROD comme B-PROD. Le modèle confond parfois la frontière entre début et continuation d'entité.

#### Visualisation 2 : Per-label F1 bar chart
**Explication intuitive** : Un graphique à barres montrant le F1 de chaque tag, permettant d'identifier d'un coup d'œil les tags problématiques.

**Application concrète** :
- Barre B-PROD : 0.92 (vert)
- Barre I-PROD : 0.88 (orange)
- Barre O : 0.965 (vert foncé)
- **Interprétation** : I-PROD est le maillon faible (F1=0.88). Les efforts d'amélioration devraient se concentrer sur la détection des tokens internes d'entités.

### 1.2.1 NERAnalyser

#### Métrique 1 : Span-level precision/recall/F1
**Explication intuitive** : Au lieu d'évaluer token par token, on évalue l'entité complète (span). Un span est correct seulement si tous ses tokens sont corrects. Un F1 span-level de 0.85 signifie que 85% des entités sont parfaitement reconnues (type + frontières exactes).

**Application concrète** : Entité "très longue batterie" (3 tokens)
- Span gold : tokens [12,13,14]
- Span prédit : tokens [12,13,14] → match parfait
- Sur 200 entités : 170 parfaitement reconnues → **F1 span-level = 0.85**
- **Interprétation** : 15% des entités ont au moins une erreur (frontière incorrecte ou type faux).

#### Métrique 2 : Partial match F1
**Explication intuitive** : Une entité partiellement correcte (ex: gold sur 3 tokens, prédit sur 2 des 3) reçoit un score partiel. Cette métrique est plus indulgente que le span-level F1. Un partial match F1 de 0.92 signifie que même si les entités ne sont pas parfaites, elles se rapprochent souvent de la vérité.

**Application concrète** :
- Gold : [12,13,14] (3 tokens), Prédit : [12,13] (2 tokens) → match partiel (2/3 = 0.67)
- Moyenne sur toutes les entités du score de chevauchement = **0.92**
- **Interprétation** : Bien que seulement 85% des entités soient parfaites, 92% ont un chevauchement substantiel avec la vérité. Les erreurs sont souvent mineures (quelques tokens manqués en fin d'entité).

#### Métrique 3 : Boundary error rate
**Explication intuitive** : Proportion d'entités où le modèle se trompe sur la position exacte du début ou de la fin (décalage de tokens). Un taux de 0.12 signifie que 12% des entités ont une frontière décalée.

**Application concrète** :
- Sur 200 entités, 24 ont un décalage de début ou fin
- **Boundary error rate** = 24/200 = **0.12**
- **Interprétation** : 12% des entités ont une frontière erronée. La plupart des erreurs sont des décalages de ±1 token.

#### Visualisation 1 : Per-entity-type F1 bar chart
**Explication intuitive** : Un graphique à barres montrant le F1 pour chaque type d'entité (PRODUIT, MARQUE, CARACTÉRISTIQUE, etc.).

**Application concrète** :
- PRODUIT : F1 = 0.88
- MARQUE : F1 = 0.92
- CARACTÉRISTIQUE : F1 = 0.76
- **Interprétation** : Les caractéristiques ("batterie", "écran", "autonomie") sont beaucoup moins bien reconnues (76%) que les produits (88%) ou marques (92%). Les caractéristiques sont plus variables et moins standardisées.

#### Visualisation 2 : Boundary error heatmap
**Explication intuitive** : Une matrice montrant la distribution des décalages de frontières (gold_start - pred_start). Elle révèle si le modèle a tendance à commencer trop tôt ou trop tard.

**Application concrète** : Matrice des décalages (gold_start - pred_start) pour le début d'entité :
```
      -2  -1   0  +1  +2
-2    0   0   2   1   0
-1    1   3  15   4   1
 0    2  18 210  12   3
+1    0   5  14   8   2
```
- Cellule (0,0) = 210 entités parfaitement alignées (décalage 0)
- Cellules (0,+1) = 12 entités où gold_start = pred_start + 1 (le modèle commence 1 token trop tard)
- Cellules (-1,0) = 15 entités où gold_start = pred_start - 1 (le modèle commence 1 token trop tôt)
- **Interprétation** : Les erreurs principales sont des décalages de +1 (12 cas) ou -1 (15 cas). Le modèle n'a pas de biais directionnel marqué.

### 1.3 PredictionDiagnosticAnalyser

#### Métrique 1 : Prediction entropy
**Explication intuitive** : L'entropie mesure l'incertitude du modèle sur sa prédiction. Plus l'entropie est élevée (max = log2(n_classes) ≈ 1.58 pour 3 classes), plus le modèle est indécis. Une entropie de 1.03 est élevée (sur une échelle de 0 à 1.58), signifiant une forte incertitude.

**Application concrète** : Commentaire "Le produit est correct sans plus"
- Distribution : [positif=0.45, neutre=0.40, négatif=0.15]
- Calcul : -[0.45*log2(0.45) + 0.40*log2(0.40) + 0.15*log2(0.15)]
- = -[0.45*(-1.15) + 0.40*(-1.32) + 0.15*(-2.74)]
- = -[-0.52 -0.53 -0.41] = 1.46 bits (sur max 1.58)
- **Interprétation** : Le modèle est très incertain (1.46/1.58 ≈ 92% de l'entropie maximale). Il hésite entre positif (45%) et neutre (40%).

#### Métrique 2 : Confidence gap between top-2 classes
**Explication intuitive** : La différence de probabilité entre la classe prédite (1ère) et la 2ème classe. Un petit écart signifie que le modèle hésite entre deux classes. Un écart de 0.05 signifie qu'il n'y a que 5 points de pourcentage de différence entre les deux premières classes.

**Application concrète** : Même commentaire "Le produit est correct sans plus"
- Top-1 = positif (0.45), Top-2 = neutre (0.40)
- **Confidence gap** = 0.45 - 0.40 = **0.05**
- **Interprétation** : Écart très faible (5 points). Le modèle est pratiquement aussi sûr du neutre que du positif. C'est un cas typique d'incertitude.

#### Métrique 3 : Per-class error rate at high-confidence threshold
**Explication intuitive** : On regarde uniquement les prédictions où le modèle est très confiant (ex: confiance > 0.9), et on calcule le taux d'erreur par classe. Cela révèle si le modèle est "sûr de se tromper" sur certaines classes. Un taux d'erreur de 0.15 pour le neutre signifie que quand le modèle prédit "neutre" avec >90% de confiance, il a tort dans 15% des cas.

**Application concrète** :
- Confiance > 0.9, classe Positif : 2% d'erreur
- Confiance > 0.9, classe Négatif : 3% d'erreur
- Confiance > 0.9, classe Neutre : 15% d'erreur
- **Interprétation** : Quand le modèle est très confiant sur le neutre, il se trompe encore dans 15% des cas. C'est grave : le modèle est victime d'illusions de compétence sur cette classe.

#### Visualisation 1 : Confidence score distribution (correct vs. wrong predictions overlaid)
**Explication intuitive** : Un histogramme superposé montrant la distribution des scores de confiance pour les prédictions correctes (une couleur) et les prédictions erronées (une autre couleur). Cela permet de voir si les erreurs surviennent plutôt à basse ou haute confiance.

**Application concrète** :
- Histogramme superposé :
  - Prédictions correctes : pic à 0.92-0.98 (confiance élevée)
  - Prédictions erronées : distribution bimodale avec pics à 0.55 et 0.85
- **Interprétation** : La plupart des erreurs à haute confiance (pic à 0.85) sont très dangereuses car le modèle est sûr de lui mais faux. Le second pic à 0.55 correspond aux cas d'incertitude normale.

## 2. GenerationAnalyser

### Métrique 1 : BLEU
**Explication intuitive** : BLEU mesure le chevauchement de n-grammes (séquences de n mots) entre le texte généré et la référence. Un score BLEU de 0.45 signifie que le texte généré partage 45% des n-grammes de la référence (pondéré). BLEU est surtout utilisé en traduction automatique.

**Application concrète** : Résumé généré vs. référence (tâche de résumé)
- Généré : "Le produit est de bonne qualité"
- Référence : "Ce produit présente une qualité satisfaisante"
- Calcul BLEU-4 (moyenne des précisions sur unigrammes à 4-grammes) = **0.42**
- **Interprétation** : Les résumés générés partagent 42% des n-grammes avec les références. C'est modéré, mais BLEU pénalise les synonymes ("bonne" vs "satisfaisante").

### Métrique 2 : ROUGE-{1,2,L}
**Explication intuitive** : ROUGE mesure le rappel des n-grammes (contrairement à BLEU qui mesure la précision). ROUGE-L regarde la plus longue sous-séquence commune. Un ROUGE-1 de 0.65 signifie que 65% des unigrammes de la référence sont présents dans le généré.

**Application concrète** : Même exemple
- ROUGE-1 (unigrammes) : 0.65
- ROUGE-2 (bigrammes) : 0.48
- ROUGE-L (plus longue séquence) : 0.60
- **Interprétation** : Le généré capture bien les mots clés (65% des unigrammes de la référence) mais moins bien les séquences de deux mots (48%). La plus longue séquence commune est "qualité" (un seul mot) car les structures sont différentes.

### Métrique 3 : BERTScore
**Explication intuitive** : BERTScore utilise des embeddings contextuels pour mesurer la similarité sémantique, pas seulement lexicale. Un BERTScore de 0.90 signifie que le généré et la référence sont très proches en termes de sens, même s'ils n'utilisent pas les mêmes mots.

**Application concrète** : Même exemple
- Généré : "Le produit est de bonne qualité"
- Référence : "Ce produit présente une qualité satisfaisante"
- BERTScore F1 = **0.90** (beaucoup plus élevé que BLEU=0.42)
- **Interprétation** : Sémantiquement, les deux phrases sont très proches. BLEU sous-estime la qualité car il ne comprend pas les synonymes. BERTScore est plus adapté pour la génération en langue naturelle.

#### Visualisation 1 : Score distribution histogram
**Explication intuitive** : Un histogramme montrant la distribution des scores BLEU/ROUGE/BERTScore sur l'ensemble des exemples. On voit la dispersion et si la performance est homogène.

**Application concrète** : Histogramme BERTScore sur 1000 résumés
- Moyenne = 0.88, médiane = 0.89, écart-type = 0.07
- Queue à gauche : 5% des résumés avec BERTScore < 0.75
- **Interprétation** : La plupart des résumés sont bons (autour de 0.89), mais 5% sont médiocres (<0.75). Ces mauvais résumés sont à analyser manuellement.

#### Visualisation 2 : Generated vs. reference length scatter plot
**Explication intuitive** : Nuage de points où chaque point est un exemple, avec en X la longueur du texte généré, en Y la longueur de la référence. La diagonale y=x représente l'égalité parfaite.

**Application concrète** :
- Nuage de points : majorité des points autour de la diagonale y=x
- Pour les résumés courts (30-50 tokens) : bien alignés
- Pour les résumés longs (80+ tokens) : points sous la diagonale → le modèle génère des résumés trop courts par rapport à la référence
- **Interprétation** : Le modèle a tendance à trop condenser les textes longs, perdant de l'information.

### 2.1 Seq2SeqAnalyser

#### Métrique 1 : METEOR
**Explication intuitive** : METEOR améliore BLEU en tenant compte des synonymes, des racines de mots (stemming), et de l'ordre des mots. Un score METEOR de 0.68 signifie une bonne correspondance sémantique, supérieure à ce que suggère BLEU.

**Application concrète** : Même exemple traduction
- BLEU = 0.42, METEOR = **0.68**
- **Interprétation** : METEOR est plus élevé car il reconnaît que "bonne" et "satisfaisante" sont sémantiquement proches (via WordNet ou synonymes).

#### Métrique 2 : chrF
**Explication intuitive** : chrF opère au niveau des caractères (n-grammes de caractères), ce qui le rend robuste aux langues à morphologie riche et aux fautes d'orthographe mineures. Un chrF de 0.83 signifie que les séquences de caractères se chevauchent à 83%.

**Application concrète** : Traduction commentaire "génial" → anglais
- Référence : "awesome" (6 caractères)
- Prédit : "great" (5 caractères)
- chrF (n=6) = F1 sur les n-grammes de caractères
  - Unigrammes : "a","w","e","s","o","m" vs "g","r","e","a","t" → chevauchement sur "e","a" → 2/6 ≈ 0.33
  - Mais chrF utilise plusieurs n et une moyenne pondérée → score final **0.83** (car les bigrammes "aw","we"... sont différents)
- **Interprétation** : chrF est plus indulgent que BLEU : "great" et "awesome" partagent des caractères communs ("e","a") mais sont différents. En pratique, pour la traduction, chrF est un bon complément.

#### Visualisation 1 : Score distribution histogram (déjà vu)

#### Visualisation 2 : Source vs. target length scatter plot
**Explication intuitive** : Pour chaque paire (source, généré), on plotte la longueur de la source (X) et la longueur du généré (Y). La pente de la régression indique le taux de compression/expansion.

**Application concrète** : Tâche de résumé

- Généré (résumé) : longueur moyenne = 35 mots
- Pente de la régression = 0.29

### 2.1.1 MachineTranslationAnalyser

#### Métrique 1 : TER (Translation Edit Rate)
**Explication intuitive** : TER mesure le nombre minimal d'éditions (insertions, suppressions, substitutions, déplacements de mots) pour transformer la traduction générée en référence, normalisé par la longueur de la référence. Un TER de 0.25 signifie qu'il faut 25 éditions pour 100 mots de référence.

**Application concrète** : Traduction "Le produit est bleu" → anglais
- Référence : "The product is blue"
- Prédit : "Product is blue" (manque "The")
- Éditions : 1 insertion ("The") sur 4 mots référence = **0.25**
- **Interprétation** : Plus TER est bas, mieux c'est. 0.25 est correct.

#### Métrique 2 : COMET (neural translation metric)
**Explication intuitive** : COMET utilise un réseau neuronal entraîné sur des évaluations humaines pour prédire la qualité d'une traduction. Un score COMET de 0.87 (typiquement entre 0 et 1) signifie une très bonne qualité, proche d'une traduction humaine.

**Application concrète** : Traduction "Ce produit est fantastique" → anglais
- Référence : "This product is fantastic"
- Prédit : "This product is great"
- COMET = **0.87**
- **Interprétation** : COMET est plus fiable que BLEU car il comprend que "fantastic" et "great" sont synonymes dans ce contexte. 0.87 est un excellent score.

#### Visualisation 1 : COMET/BLEU score vs. source length scatter plot
**Explication intuitive** : Nuage de points montrant comment la qualité de la traduction varie avec la longueur de la source.

**Application concrète** :
- Axe X : longueur source (mots)
- Axe Y : COMET
- Nuage : points élevés (0.85-0.95) pour sources courtes (<20 mots)
- Dégradation progressive : pour sources >50 mots, COMET descend à 0.65-0.75
- **Interprétation** : Le modèle traduit bien les phrases courtes, mais sa qualité chute sur les phrases longues (problème de contexte).

#### Visualisation 2 : Lexical error distribution
**Explication intuitive** : Diagramme à barres montrant la proportion de chaque type d'erreur (substitution, insertion, délétion, déplacement) dans les erreurs de traduction.

**Application concrète** : Analyse sur 1000 traductions
- Substitutions (mauvais mot) : 45%
- Insertions (mot en trop) : 12%
- Délétions (mot manquant) : 8%
- Déplacements (ordre des mots) : 35%
- **Interprétation** : Le principal problème est la substitution (presque la moitié des erreurs), suivie par l'ordre des mots (35%). Les insertions/délétions sont rares.

### 2.1.2 SummarizationAnalyser

#### Métrique 1 : Compression ratio
**Explication intuitive** : Rapport entre la longueur du résumé et la longueur du texte source. Un ratio de 0.25 signifie que le résumé est 4 fois plus court que l'original.

**Application concrète** : Commentaire de 200 mots → résumé de 50 mots
- **Compression ratio** = 50/200 = **0.25**
- **Interprétation** : Le modèle condense l'information à 25% de la longueur originale.

#### Métrique 2 : Extractive density
**Explication intuitive** : Proportion de mots dans le résumé qui sont copiés mot pour mot du texte source (vs. reformulés). Une densité extractive de 0.45 signifie que 45% du résumé est copié directement.

**Application concrète** : Résumé "La batterie tient longtemps, bonne autonomie"
- Source : "La batterie de ce téléphone tient très longtemps, ce qui offre une excellente autonomie"
- Mots copiés : "La batterie", "longtemps", "autonomie" → 3 mots sur 6 du résumé = 0.50
- **Extractive density** = **0.50**
- **Interprétation** : La moitié du résumé est extractive (copiée), l'autre moitié est abstraite (reformulée). Équilibre sain.

#### Métrique 3 : Coverage
**Explication intuitive** : Proportion des faits clés du texte source qui sont présents dans le résumé. Un coverage de 0.67 signifie que 2 faits sur 3 sont couverts.

**Application concrète** : Commentaire smartphone de 200 mots avec 3 faits clés :
1. Batterie longue durée
2. Écran HD
3. Prix élevé
Résumé : mentionne (1) et (2) mais pas (3)
- **Coverage** = 2/3 = **0.67**
- **Interprétation** : Le résumé omet un fait important (prix élevé). Coverage complète la compression ratio.

#### Métrique 4 : Faithfulness score (via NLI)
**Explication intuitive** : Score mesurant si le résumé contient des informations non présentes dans la source (hallucinations). Un score de 0.92 signifie que 92% des affirmations du résumé sont vérifiables dans la source.

**Application concrète** : Résumé "L'écran est pliable et la batterie dure 2 jours"
- Source : mentionne la batterie (vrai) mais ne mentionne pas d'écran pliable (faux)
- Modèle NLI : prédit "contradiction" pour "écran pliable" → pénalité
- **Faithfulness score** = 0.85 (car une hallucination sur deux)
- **Interprétation** : Le modèle hallucine parfois (15% des cas). C'est un problème pour les applications critiques.

### 2.2 LLMJudgeAnalyser

#### Métrique 1 : Per-criterion score (relevance, coherence, faithfulness)
**Explication intuitive** : Score (ex: 1-5) attribué par un LLM (ex: GPT-4) pour chaque critère. Une pertinence de 4.2/5 signifie que le LLM juge la réponse globalement pertinente mais avec une petite réserve.

**Application concrète** : Évaluation d'une réponse RAG
- Pertinence : 4.5/5
- Cohérence : 4.0/5
- Fidélité : 3.8/5
- **Interprétation** : La réponse est pertinente (bonne) et cohérente (correcte), mais la fidélité est un peu faible (peut-être une légère hallucination).

#### Métrique 2 : Calibrated score (Rogan & Gladen correction)
**Explication intuitive** : Correction des biais systématiques du LLM juge (ex: tendance à favoriser les réponses longues ou en première position). Un score brut de 4.2 devient 4.05 après calibration.

**Application concrète** :
- Score brut moyen = 4.2/5
- Biais identifié : le LLM juge note systématiquement 0.15 plus haut les réponses longues (>100 mots)
- Score calibré = 4.2 - 0.15 = **4.05**
- **Interprétation** : Après correction du biais, le vrai score est 4.05.

#### Métrique 3 : Self-consistency score
**Explication intuitive** : Variance des scores quand on appelle le LLM juge plusieurs fois (N fois) sur le même input. Une variance de 0.1 signifie que le juge n'est pas parfaitement cohérent avec lui-même.

**Application concrète** : 10 appels du LLM sur le même exemple
- Scores : [4.2, 4.1, 4.3, 4.0, 4.2, 4.1, 4.2, 4.3, 4.0, 4.2]
- Moyenne = 4.16, écart-type = 0.09 → variance = **0.008** (très faible)
- **Interprétation** : Le LLM juge est très cohérent avec lui-même (faible variance). On peut lui faire confiance.

#### Métrique 4 : Inter-judge agreement (Cohen's κ)
**Explication intuitive** : Accord entre deux LLM juges différents (ex: GPT-4 et Claude). κ=0.82 signifie un accord presque parfait.

**Application concrète** : GPT-4 vs. Claude sur 100 évaluations
- Accord observé = 89%
- Accord dû au hasard = 40%
- κ = (0.89-0.40)/(1-0.40) = 0.49/0.60 = **0.82**
- **Interprétation** : κ > 0.80 → accord presque parfait. Les deux LLM jugent de manière similaire.

#### Visualisation 1 : Per-criterion radar chart
**Explication intuitive** : Graphique radar (en toile d'araignée) montrant les scores pour chaque critère, permettant de voir visuellement les forces/faiblesses.

**Application concrète** :
- Axes : Pertinence, Cohérence, Fidélité, Fluidité, Utilité
- Valeurs : 4.5, 4.0, 3.8, 4.2, 4.1
- Forme : légèrement aplatie sur l'axe Fidélité (3.8)
- **Interprétation** : La fidélité est le point faible. Le radar le montre instantanément.

#### Visualisation 2 : Raw vs. calibrated score distribution per judge
**Explication intuitive** : Boîtes à moustaches ou histogrammes comparant la distribution des scores bruts et calibrés pour chaque juge.

**Application concrète** :
- GPT-4 brut : médiane 4.2, IQR [4.0-4.4]
- GPT-4 calibré : médiane 4.05, IQR [3.85-4.25] (décalage vers le bas)
- **Interprétation** : La calibration a décalé toute la distribution de -0.15.

### 2.3 AbstractiveQAAnalyser

#### Métrique 1 : ROUGE-L (avec référence)
**Explication intuitive** : Version de ROUGE utilisant la plus longue sous-séquence commune. Un ROUGE-L de 0.60 signifie que 60% de la séquence de la réponse de référence est retrouvée dans la réponse générée.

**Application concrète** : QA sur article de produit
- Question : "Quelle est la capacité de la batterie ?"
- Réponse générée : "5000 mAh"
- Référence : "La batterie a une capacité de 5000 milliampères-heures"
- Plus longue sous-séquence commune : "5000 mAh" (2 tokens) sur référence (9 tokens) → ROUGE-L = 2/9 ≈ 0.22 (très faible)
- **Interprétation** : ROUGE-L sous-estime la qualité car la réponse est correcte mais plus courte. Pas adapté pour QA.

#### Métrique 2 : BERTScore (avec référence)
**Explication intuitive** : Même principe que pour la génération, mais pour la QA. Un BERTScore de 0.89 signifie une forte similarité sémantique avec la référence.

**Application concrète** : Même exemple
- BERTScore F1 = **0.89**
- **Interprétation** : Sémantiquement, "5000 mAh" et "5000 milliampères-heures" sont quasi identiques. BERTScore est bien plus adapté que ROUGE pour la QA.

#### Métrique 3 : Faithfulness (via LLMJudge, sans référence)
**Explication intuitive** : Le LLM juge si la réponse est fidèle au document source (pas d'hallucination). Un score de 0.85 signifie que la réponse est fidèle à 85%.

**Application concrète** :
- Document source : "La batterie du X100 a une capacité de 5000mAh"
- Question : "Quelle est la capacité de la batterie ?"
- Réponse : "5000 mAh" → faithfulness = 0.98
- Réponse : "4500 mAh" → faithfulness = 0.45 (contradiction)
- **Interprétation** : Le LLM juge détecte bien les hallucinations. La version avec référence (BERTScore) n'est pas toujours disponible.

#### Métrique 4 : Answer relevance (via LLMJudge, sans référence)
**Explication intuitive** : Le LLM juge si la réponse répond réellement à la question (peut être fidèle mais hors-sujet). Un score de 0.75 signifie que la réponse est partiellement pertinente.

**Application concrète** :
- Question : "Quelle est la capacité de la batterie ?"
- Réponse : "Le X100 a un écran HD" → fidèle au document mais hors-sujet → relevance = 0.10
- Réponse : "5000 mAh" → relevance = 0.95
- **Interprétation** : Pertinence et fidélité sont deux dimensions différentes. Il faut les deux.

## 3. ExtractiveQAAnalyser

### Métrique 1 : Exact Match (EM)
**Explication intuitive** : Proportion de réponses qui correspondent exactement à la réponse attendue (caractère pour caractère). Un EM de 0.65 signifie que 65% des réponses sont parfaites.

**Application concrète** : Contexte = "Le X100 sorti en 2023 a une batterie 5000mAh"
- Question : "Quand est sorti le X100 ?"
- Span gold : "2023"
- Span prédit : "2023" → EM = 1.0
- Span prédit : "2023." (avec point) → EM = 0 (car caractère différent)
- Sur 1000 questions, 650 réponses exactes → **EM = 0.65**
- **Interprétation** : EM est très strict (pénalise la ponctuation). 65% de réponses parfaites.

### Métrique 2 : Token-level F1
**Explication intuitive** : F1 au niveau des tokens (mots), plus indulgent que EM. Un token F1 de 0.82 signifie que les réponses se chevauchent à 82% en moyenne avec la réponse gold.

**Application concrète** : Même exemple
- Gold : "2023" (1 token)
- Prédit : "2023." (1 token, mais "." compte comme token séparé ? Selon tokenisation)
- Si tokenisation sur mots : "2023" et "2023." sont différents → précision=0, rappel=0 → F1=0
- Si tokenisation inclut ponctuation : "2023" ≠ "2023." → toujours 0
- Donc pour une réponse d'un token, EM = F1.
- Pour une réponse plus longue : Gold "2023 et 2024", Prédit "2023" → F1=0.67
- **Interprétation** : F1 est plus informatif pour les réponses multi-tokens.

### Métrique 3 : Answer position accuracy
**Explication intuitive** : Précision de la position relative de la réponse dans le contexte (offset). Un score élevé indique que le modèle localise bien la réponse, même si le texte exact diffère légèrement.

**Application concrète** : Contexte de 1000 caractères
- Position gold : caractères 200-203 ("2023")
- Position prédite : caractères 200-204 ("2023.") → décalage de 1
- Answer position accuracy = 1 - (décalage_absolu / longueur_contexte) = 1 - (1/1000) = 0.999
- **Interprétation** : Le modèle a presque parfaitement localisé la réponse, même si le span exact est un peu différent.

#### Visualisation 1 : EM and F1 distribution histogram
**Explication intuitive** : Histogramme superposé des scores EM et F1.

**Application concrète** :
- EM : pic à 0 (40% des exemples) et pic à 1 (30% des exemples) → distribution bimodale
- F1 : distribution plus étalée, moyenne 0.65
- **Interprétation** : Beaucoup d'échecs complets (EM=0) mais aussi beaucoup de réussites parfaites (EM=1). Peu de cas intermédiaires.

#### Visualisation 2 : Answer span length distribution
**Explication intuitive** : Histogramme de la longueur des réponses (gold).

**Application concrète** :
- 60% des réponses : 1 token
- 25% : 2-3 tokens
- 10% : 4-5 tokens
- 5% : 6+ tokens
- **Interprétation** : La majorité des réponses sont courtes (1 token). Le modèle doit exceller sur les réponses d'un seul mot.

#### Visualisation 3 : F1 score vs. context length scatter plot
**Explication intuitive** : Nuage de points montrant comment le F1 varie avec la longueur du contexte.

**Application concrète** :
- Contexte < 500 tokens : F1 moyen = 0.85
- Contexte 500-1000 tokens : F1 = 0.72
- Contexte > 1000 tokens : F1 = 0.58
- Régression linéaire : pente = -0.0003 par token
- **Interprétation** : Dégradation linéaire significative. Le modèle a du mal à localiser la réponse dans les longs contextes.

### 5.1 ASRAnalyser

#### Métrique 1 : Word Error Rate (WER)
**Explication intuitive** : Proportion de mots erronés (substitutions + insertions + déletions) par rapport à la référence. 0.20 signifie 20% d'erreurs (80% correct).

**Application concrète** : Transcription "la batterie tient deux jours"
- Référence : "la batterie tient deux jours"
- Prédit : "la batterie tient de jours"
- Substitutions : "deux"→"de" (1 erreur), Insertions : 0, Délétions : 0
- Total mots référence = 5
- WER = (1+0+0)/5 = **0.20**
- **Interprétation** : 80% des mots sont corrects. Une erreur sur "deux".

#### Métrique 2 : Character Error Rate (CER)
**Explication intuitive** : Même principe que WER mais au niveau des caractères. Plus strict. Un CER de 0.08 signifie 8% de caractères erronés.

**Application concrète** : Même exemple
- Référence : "deux" (4 caractères)
- Prédit : "de" (2 caractères)
- Caractères corrects : "d","e" (2), manquants : "u","x" (2)
- CER sur ce mot = 2/4 = 0.50
- CER moyen sur toute la phrase = **0.08** (car une seule erreur sur beaucoup de caractères)
- **Interprétation** : CER est plus indulgent que WER pour les petites erreurs.

#### Métrique 3 : WER stratified by speaker and utterance length
**Explication intuitive** : WER calculé séparément pour différents locuteurs et différentes longueurs d'utterance.

**Application concrète** :
- Locuteur A (homme, débit rapide) : WER=0.25
- Locuteur B (femme, débit lent) : WER=0.12
- Utterances < 5 mots : WER=0.05
- Utterances 5-10 mots : WER=0.15
- Utterances > 10 mots : WER=0.30
- **Interprétation** : Le modèle performe moins bien sur les voix rapides et les phrases longues.

#### Métrique 4 : Real-Time Factor (RTF, optionnel)
**Explication intuitive** : Rapport entre le temps de traitement et la durée de l'audio. Un RTF de 0.5 signifie que le modèle traite 2 secondes d'audio par seconde réelle.

**Application concrète** :
- Durée audio = 10 secondes
- Temps de transcription = 5 secondes
- RTF = 5/10 = **0.5**

## 6. FillMaskAnalyser

### Métrique 1 : Top-k accuracy (k ∈ {1, 5, 10})
**Explication intuitive** : Proportion de fois où le token masqué est dans le top-k des prédictions. Une top-1 accuracy de 0.72 signifie que dans 72% des cas, le modèle prédit exactement le bon mot.

**Application concrète** : Phrase "La [MASK] du X100 dure 2 jours"
- Token gold : "batterie"
- Top-1 prédit : "batterie" → top-1 = 1.0
- Top-5 prédit : ["batterie", "autonomie", "charge", "pile", "alimentation"] → top-5 = 1.0
- Sur 1000 phrases : top-1 = 0.72, top-5 = 0.89, top-10 = 0.94
- **Interprétation** : Le modèle a raison dans 72% des cas (top-1). Si on accepte les 5 meilleurs, on monte à 89%.

### Métrique 2 : MRR (Mean Reciprocal Rank) over vocabulary candidates
**Explication intuitive** : Moyenne des inverses du rang du token correct. Un MRR de 0.78 signifie que le token correct est en moyenne à la position 1/0.78 ≈ 1.28 (donc souvent en 1ère position).

**Application concrète** :
- Exemple 1 : gold = "batterie", rang=1 → reciprocal rank=1
- Exemple 2 : gold = "autonomie", rang=2 → reciprocal rank=0.5
- Exemple 3 : gold = "charge", rang=3 → reciprocal rank=0.33
- MRR = (1 + 0.5 + 0.33)/3 = 1.83/3 = **0.61**
- **Interprétation** : MRR est plus sévère que top-1 : il pénalise les rangs élevés.

### Métrique 3 : Pseudo-log-likelihood (PLL)
**Explication intuitive** : Log-vraisemblance du modèle sur le contexte (pas une métrique de prédiction). Un PLL plus élevé signifie que le modèle est plus "confiant" dans le contexte global.

**Application concrète** :
- Phrase : "La batterie est excellente"
- PLL = -2.3 (log probabilité)
- **Interprétation** : À interpréter séparément du top-k accuracy. Un PLL élevé n'implique pas une bonne prédiction.

#### Visualisation 1 : Top-k predicted token probability bar chart
**Explication intuitive** : Diagramme à barres des probabilités du top-k.

**Application concrète** :
- "batterie" : 0.67
- "autonomie" : 0.18
- "charge" : 0.07
- "pile" : 0.04
- "alimentation" : 0.02
- **Interprétation** : Distribution très concentrée sur "batterie" (67%). Peu d'ambiguïté.

#### Visualisation 2 : Accuracy breakdown by grammatical category
**Explication intuitive** : Top-1 accuracy séparée par catégorie grammaticale du token masqué.

**Application concrète** :
- Noms : 0.81
- Verbes : 0.68
- Adjectifs : 0.62
- Déterminants : 0.95
- **Interprétation** : Le modèle excelle sur les déterminants (souvent prévisibles), moins sur les adjectifs (plus variables).

#### Visualisation 3 : MRR distribution across the test set
**Explication intuitive** : Histogramme des MRR individuels (par exemple).

**Application concrète** :
- Pic à 1.0 (30% des exemples ont rang=1)
- Queue jusqu'à 0.1 (rang=10)
- **Interprétation** : 30% des tokens sont parfaitement prédits (rang 1). Le reste a des rangs variables.

## 7. RetrievalAnalyser

### Métrique 1 : Precision@k
**Explication intuitive** : Proportion de documents pertinents parmi les k premiers retrievés. Une Precision@5 de 0.60 signifie que sur les 5 premiers documents, 3 sont pertinents.

**Application concrète** : Query "batterie X100", k=5
- Documents retrievés : [pertinent, non, pertinent, non, pertinent] → 3 pertinents
- Precision@5 = 3/5 = **0.60**
- **Interprétation** : 60% des documents retrievés sont utiles.

### Métrique 2 : Recall@k
**Explication intuitive** : Proportion des documents pertinents totaux qui sont dans les k premiers retrievés. Un Recall@5 de 0.75 signifie que 75% de tous les documents pertinents sont dans le top-5.

**Application concrète** : Total documents pertinents = 4 (sur 1000)
- Top-5 contient 3 de ces 4 → Recall@5 = 3/4 = **0.75**
- **Interprétation** : On trouve 75% des documents pertinents dans les 5 premiers.

### Métrique 3 : NDCG@k (Normalized Discounted Cumulative Gain)
**Explication intuitive** : Prend en compte le rang et la pertinence graduée (ex: 0=non pertinent, 1=peu, 2=très pertinent). Un NDCG@5 de 0.985 est presque parfait.

**Application concrète** : Pertinence graduée [3, 2, 0, 1, 0] (3=parfait)
- DCG@5 = 3/log2(2) + 2/log2(3) + 0 + 1/log2(5) = 3 + 1.26 + 0 + 0.43 = 4.69
- IDCG@5 (ordre idéal) : [3,2,1,0,0] → 3/log2(2)+2/log2(3)+1/log2(4)=3+1.26+0.5=4.76
- NDCG@5 = 4.69/4.76 = **0.985**
- **Interprétation** : Le ranking est presque optimal (petite erreur sur le document de pertinence 1).

### Métrique 4 : MRR (Mean Reciprocal Rank)
**Explication intuitive** : Moyenne de l'inverse du rang du premier document pertinent. Un MRR de 0.90 signifie qu'en moyenne, le premier pertinent est au rang 1/0.90 ≈ 1.11 (souvent en 1ère position).

**Application concrète** :
- Query 1 : premier pertinent au rang 1 → 1/1=1.0
- Query 2 : premier pertinent au rang 2 → 1/2=0.5
- Query 3 : premier pertinent au rang 1 → 1.0
- MRR = (1.0+0.5+1.0)/3 = 2.5/3 = **0.833**
- **Interprétation** : Le plus souvent, le premier pertinent est en position 1, parfois en position 2.

### Métrique 5 : MAP (Mean Average Precision)
**Explication intuitive** : Moyenne des précisions à chaque rang où un document pertinent apparaît. Un MAP de 0.75 signifie une bonne performance.

**Application concrète** : Query avec pertinences [1,0,1,1,0] (3 pertinents)
- Précision@1 = 1/1=1.0
- Précision@3 = 2/3=0.67 (2 pertinents sur 3)
- Précision@4 = 3/4=0.75
- Average Precision = (1.0+0.67+0.75)/3 = 2.42/3 = 0.81
- MAP = moyenne sur toutes les queries = **0.75**
- **Interprétation** : En moyenne, la précision sur les rangs pertinents est de 75%.

#### Visualisation 1 : Precision@k and Recall@k vs. k curves
**Explication intuitive** : Courbes montrant comment précision et rappel évoluent avec k.

**Application concrète** :
- P@1=0.9, P@5=0.7, P@10=0.5, P@20=0.3
- R@1=0.2, R@5=0.6, R@10=0.8, R@20=0.9
- **Interprétation** : Compromis classique : plus on augmente k, plus la précision baisse et le rappel monte.

#### Visualisation 2 : Rank distribution of first relevant document
**Explication intuitive** : Histogramme du rang du premier document pertinent.

**Application concrète** :
- Rang 1 : 60% des queries
- Rang 2 : 20%
- Rang 3 : 10%
- Rang >3 : 10%
- **Interprétation** : Dans 60% des cas, le meilleur document est en première position. Dans 10% des cas, il est au-delà du rang 3 (problème).

#### Visualisation 3 : Query-level NDCG distribution histogram
**Explication intuitive** : Histogramme des NDCG individuels par query.

**Application concrète** :
- Moyenne = 0.85, médiane = 0.88
- 10% des queries avec NDCG < 0.60 (mauvais ranking)
- **Interprétation** : La plupart des queries sont bien traitées, mais 10% sont problématiques.
